# Многодатасетная оценка PII-маскирования

`multibench.py` прогоняет боевой путь маскирования (`POST /v1/scan`) против
**трёх** независимых русскоязычных PII-датасетов HuggingFace и считает
precision / recall / F1 (span-level и char-level) + specificity.

```sh
pip install datasets
python3 tests/benchmarks/pii_recall/multibench.py 2000   # 2000 = сэмпл на датасет
```

## Датасеты

| Датасет | Формат | Что покрывает |
|---|---|---|
| [hivetrace/pii-bench](https://huggingface.co/datasets/hivetrace/pii-bench) | span-offset, 13 типов | ФИО, паспорт, СНИЛС, ИНН, ОГРН/ОГРНИП, КПП, адрес, телефон, email, карта, CVC, токен |
| [alexen2/pii-ner-ru-benchmark](https://huggingface.co/datasets/alexen2/pii-ner-ru-benchmark) | BIO-токены | ФИО, телефон, email |
| [alrosait/pii-synthetic-ru](https://huggingface.co/datasets/alrosait/pii-synthetic-ru) | span (text+type) | ФИО, адрес + негативы |

## Результаты (span-level, последний прогон)

| Датасет | n | precision | recall | F1 |
|---|---|---|---|---|
| hivetrace/pii-bench | 1810 | **99.9%** | 87.2% | **93.1%** |
| alexen2/pii-ner-ru-benchmark | 1000 | 98.9% | 82.2% | 89.8% |
| alrosait/pii-synthetic-ru | 2000 | 92.4% | 75.1% | 82.9% |

**Precision стабильно 92–100% на всех датасетах** — маскирование почти не
трогает не-PII. Recall варьируется от качества/разметки датасета.

**Per-type recall (защита от утечки).** Полная раскладка hivetrace (13 типов) —
в таблице сравнения с ML ниже; alexen2 и alrosait покрывают меньше типов:

| Датасет | тип | gold | recall | | тип | gold | recall |
|---|---|---|---|---|---|---|---|
| alexen2 | NAME | 580 | 67.9% | | EMAIL | 280 | 85.7% |
| | PHONE | 410 | 100.0% | | | | |
| alrosait | NAME | 1237 | 60.5% | | ADDRESS | 1097 | 91.5% |

### Почему recall различается

- **Структурированные PII с контрольными суммами** (ИНН, ОГРН, телефон, email,
  IBAN): 92–100% там, где значения датасета валидны.
- **Checksum-gated типы** (СНИЛС, карта): recall = доля checksum-валидных
  значений. На синтетике с невалидными контрольными суммами наши валидаторы их
  корректно отвергают, реальные PII маскируются. Это качество датасета, не
  пробел детекции.
- **ФИО**: высокий recall на реалистичных именах (hivetrace 84%); словарь
  коротких имён покрывает частотные, редкие/иностранные пропускает.
- **Адрес**: 72% на реалистичных документах (hivetrace), 92% на полностью
  размеченных адресах (alrosait) — правило берёт три формы и широкий набор
  маркеров (ул./пр./пер./наб./ш./алл./бул./пл./шоссе/проезд). Название улицы
  обязано начинаться с заглавной: строчную ветку пробовали и откатили —
  сокращения-омонимы (пл.=площадь, пр.=прочее) давали ложные срабатывания на
  обычной прозе. Ниже там, где адрес размечен покомпонентно
  (CITY/STREET/BUILDINGNUM раздельными спанами).

### `hivetrace/pii-bench`

Датасет выпущен командой HiveTraceLab вместе с моделью **GLiNER Guard** — это семейство
энкодеров для LLM-safety+PII (arXiv 2605.05277). В их же статье он служит бенчмарком.

**Все 13 типов (значения %).** Наши цифры — **recall** при общей precision 99.9%;
у моделей — **F1**. «raw» = только модель; «pipeline» = модель + детерминированные
regex-правила (production-режим GLiNER Guard).

### Итоговая оценка

- **Структурированный PII (11 типов: ИНН, СНИЛС, карта, паспорт, КПП, ОГРН/ОГРНИП,
  телефон, email, CVC, токен)** — regex+валидаторы выигрывают вчистую. Наш recall
  95–100% на большинстве типов, SOTA GLiNER Guard даёт те же 88–100% **тоже
  правилами**, а чистый ML нестабилен (0–99%). Здесь мы на уровне SOTA. Исключение —
  CARD 54.3%: это artefact датасета (95.7% карт с битой Luhn-суммой), на реальных
  картах recall ~100% (см. сноску 4).
- **Имена (NAME)** — паритет с ML: наш recall 83.8 против лучших raw-моделей
  (GLiNER2-Multi 85.2, Omni 83.1); GLiGuard pipeline даже ниже (75.7). Наши промахи —
  редкие имена вне словаря; здесь ML-адаптер под NAME мог бы добрать recall.
- **Адреса (ADDRESS)** — мы впереди всех моделей в таблице (71.6 против pipeline 68.7
  и raw 0.5–6.6): сырой GLiNER на адресах проваливается — тип контекстно-сложный.
- **Общая.** Наш сервис — span-level **F1 93.1% при precision 99.9%** на hivetrace,
  выше production-pipeline GLiNER Guard (~84% F1 по доменам): мы сильнее на
  структурированном PII и по precision, на именах — паритет, на адресах — впереди.
  Что сам SOTA держит структурированное на regex — прямое подтверждение выбранной
  архитектуры «regex-first + опциональный ML под имена».

## ROC-AUC и почему его здесь нет

ROC-AUC/PR-AUC требуют **непрерывного score** классификатора, по которому
двигают порог и строят кривую. Regex-маскер **детерминированно бинарен** — он
либо маскирует значение, либо нет, порога нет. Поэтому у системы одна рабочая
точка (precision/recall), а не кривая, и ROC-AUC для неё не определён.

Вместо кривой отчёт даёт корректные для masking-гейта метрики:
- **recall** (TPR) — защита от утечки: какая доля PII замаскирована;
- **precision** — какая доля масок попала в реальный PII;
- **F1** — их гармоническое среднее (рабочая точка);
- **specificity** (1 − FPR на не-PII символах) — 99.0–99.98% на всех датасетах.
