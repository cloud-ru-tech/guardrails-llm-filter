# Настройки, модель доверия и логирование

## Семантика резолюции настроек

Три слоя, от статического к пофазовому:

1. **Env-дефолты** (`GUARDRAILS_ENABLED`, `GUARDRAILS_MODE`, `GUARDRAILS_DATA_TYPES`)
   парсятся на старте (плохие значения паникуют на старте) и становятся начальным
   in-process-кэшем.
2. **Хранилище** (`settings.Service`): `Load()` на старте читает хранилище; если ничего
   не было сохранено — **засевает хранилище env-дефолтами** (чтобы API всегда
   читал/писал один источник истины). Далее `PUT /v1/settings` → `Update()` (запись +
   атомарная замена кэша), а тикер `RunRefresh` перечитывает хранилище, чтобы другие
   реплики сходились. Важное следствие: **менять env-переменные после первого старта
   бесполезно**, пока в хранилище лежит сохранённое значение (кроме бэкенда in_memory,
   который забывает при рестарте).
3. **Override-заголовок запроса** — `settings.Effective(global, headerValue)`, чистая
   функция:
   - пустой заголовок → применяются глобальные настройки;
   - `none` → guardrails пропускаются для запроса;
   - CSV из чисел/имён → **пересечение** с глобальными типами данных (только сужение;
     нельзя включить то, что глобально выключено);
   - пустое пересечение → отключено для запроса;
   - любой неразбираемый токен → весь заголовок игнорируется, действует полная глобальная
     защита (fail toward protection).

**Семантика режима** (`mode` в документе настроек, `enforce`/`detect`): только
глобальная — override-заголовок может сузить типы данных или пропустить запрос, но
никогда не переключает режим. В `detect` (shadow) пайплайн сканирует, увеличивает метрики
срабатываний и `requests_masked_total{mode="detect"}`, пишет аудит-записи (включите
`GUARDRAILS_AUDIT_ENABLED`), но тело не изменяется, masking state не сохраняется и фаза
ответа не демаскирует. Невалидное/отсутствующее значение нормализуется к `enforce`.
`PUT /v1/settings` заменяет весь документ: пропуск `mode` сбрасывает его в `enforce`.

## Модель доверия к override-заголовку

Заголовок — **доверенный ввод**, не клиента. Сервис потребляет его и не форвардит
upstream. Если сервис доступен недоверенным клиентам, фронтящий шлюз должен вырезать
заголовок из клиентских запросов, чтобы вызывающая сторона не сужала собственный объём
маскирования. Семантика «только сужение» ограничивает урон от подделанного заголовка до
«меньше проверок на одном запросе»; фичу можно полностью отключить
(`GUARDRAILS_OVERRIDE_HEADER=""`).

## Доверие к `x-request-id`

`x-request-id` — ключ masking state. При одиночной реплике запрос и ответ обрабатываются
в одном процессе, и состояние живёт в памяти. С общим хранилищем (`redis`/`postgres`) и
несколькими репликами фаза ответа может откатиться к `GetMaskingState(x-request-id)`,
если запрос обработала другая реплика. В этом случае `x-request-id` должен задаваться
доверенной стороной (сервисом или фронтящим шлюзом), а не приниматься от
неаутентифицированного клиента: клиент, способный подделать живой request id жертвы (плюс
литеральный текст плейсхолдера в своём теле), мог бы демаскировать оригиналы жертвы в свой
ответ. См. [../storage/](../storage/) и [../operations/](../operations/).

## Логирование

`internal/logging`: `Setup(level, format)` ставит default для slog (JSON- или text-хендлер,
`AddSource: true`). Хелперы повторяют slog, но `Error`/`Fatal` берут ошибку позиционно и
терпимы к nil. Поля лога на область запроса — path, request_id, model, sse — дописываются
в местах вызова.
