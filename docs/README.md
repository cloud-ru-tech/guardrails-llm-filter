который маскирует PII и секреты в трафике к LLM и демаскирует ответы.
Пользовательский обзор — в корневом [README.md](../README.md).

## Коротко о сервисе

Сервис стоит между вашими клиентами и LLM-провайдером: клиенты направляют запросы в
него вместо провайдера. На запросе он сканирует JSON-тела OpenAI/Anthropic набором из
~260 regex-правил и заменяет найденные значения плейсхолдерами (`<EMAIL_1>`), затем сам
пересылает маскированный запрос провайдеру и восстанавливает оригиналы в ответе — в том
числе внутри SSE-потоков и аргументов tool-call. Envoy или сайдкара в data-path нет:
guardrails-llm-filter сам является data-plane. (Родственный проект
`guardrails-llm-filter-extproc` упаковывает тот же движок как gRPC-сайдкар Envoy
`ext_proc`.)

Каждое изменение подчиняется двум инвариантам: модель вердикта — **только
маскировать/пропускать** (запрос никогда не блокируется), а data-path работает
**fail-open** (любая внутренняя ошибка — сбой хранилища, ошибка маскирования,
некорректное правило — пропускает трафик, а не ломает его). Глобальная политика живёт в
подключаемом хранилище (`in_memory`/`redis`/`postgres`), меняется на лету через
management API и может быть только **сужена** для конкретного запроса доверенным
заголовком.
